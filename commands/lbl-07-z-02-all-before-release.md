# 需求收尾全流程

需求开发完成后，按顺序执行以下 10 个步骤，每步完成后再进入下一步。中途发现问题直接修复，不要跳步。

## 参数

$ARGUMENTS — 功能分支名、功能目录路径、或功能名称

## 文件编号规则

所有输出文件放在 `myfeature/功能目录/output/` 下，编号从 `02-` 开始，与 `/lbl-01`（头脑风暴）生成的 `01-` 文件衔接。

### 需求编号提取规则

从功能目录名中匹配 `YYMMDD-NNNN-功能名` 格式，取 `NNNN` 部分作为需求编号。例如目录名 `260429-2424-订阅文档更新通知` → 需求编号为 `2424`。

所有输出文件名格式统一为 `编号-<需求编号>-文件名`：

| 编号 | 来源 | 文件名 |
|------|------|--------|
| 01 | `/lbl-01`（头脑风暴） | `01-<需求编号>-design.md`、`01-<需求编号>-plan.md` |
| 02 | 本流程 步骤 2 | `02-<需求编号>-日志记录.md` |
| 03 | 本流程 步骤 3 | `03-<需求编号>-后端功能点对照.md` |
| 04 | 本流程 步骤 4 | `04-<需求编号>-<功能名>-建表SQL.sql` |
| 05 | 本流程 步骤 5 | `05-<需求编号>-test-api.js` |
| 06 | 本流程 步骤 6 | `06-<需求编号>-<功能名>-前端接口文档.md` |
| 07 | 本流程 步骤 8 | `07-<需求编号>-部署及验证.md` |
| 08 | 本流程 步骤 9 | `08-<需求编号>-会话记忆.md` |
| 09 | 本流程 步骤 10 | `09-<需求编号>-需求对照.md` |

> 步骤 7（代码审查）不生成独立编号文件，审查问题直接修复。

### ⚠️ 文件命名强制规则

**每个子 skill 有自己的文件命名规则，本表必须与子 skill 保持一致。** 执行子 skill 时，必须使用子 skill 定义的完整文件名格式，不得简化或省略功能名：

- `/lbl-04-gen-sql` 规定文件名为 `04-功能名-建表SQL.sql` → 必须带功能名
- `/lbl-06-frontend-api-doc` 规定文件名为 `06-功能名-前端接口文档.md` → 必须带功能名
- `<功能名>` 用中文，如"同步推送开关"、"模板排序"

## 步骤 2/10：生成日志记录文档

扫描分支上所有新增的日志语句（后端 `Log::` / `log.`，前端 `console.`），生成 `02-日志记录.md`。

### 执行方法

1. 找到功能分支与基准分支（release / master）的 merge-base
2. `git diff <merge-base>..<branch>` 提取所有新增的日志行（`^\+.*Log::|^\+.*console\.`）
3. 对每条日志，定位所在文件和行号（从分支最新代码 `git show <branch>:<file>` 中 grep）
4. 按 后端/前端 分组，每个文件一个子表格

### 输出文件

`myfeature/功能目录/output/02-<需求编号>-日志记录.md`

### 输出格式

```markdown
# 02-<需求编号>-日志记录

> 分支: `分支名`（涉及的子项目）
> 统计日期: YYYY-MM-DD

## 汇总

| 端 | 新增日志数 |
|---|---|
| 后端 | N 条 |
| 前端 | M 条 |
| **合计** | **X 条** |

## 后端日志（N 条）

### 文件名（K 条）

| # | 行号 | 级别 | 标签 | 描述 | 携带参数 |
|---|---|---|---|---|---|
| 1 | 123 | info | `[lbl][...][Save] done` | 保存完成 | id, count |

## 前端日志（M 条）

### 文件名（K 条）

| # | 行号 | 级别 | 标签 | 描述 |
|---|---|---|---|---|
| 1 | 45 | warn | `[Module] xxx failed` | 操作失败 |

## 日志命名规范

- 后端前缀 / 前端前缀 / 级别使用说明
```

## 步骤 7/10：代码审查

执行 `/lbl-07-review` skill。

### 审查后自动修复

审查完成后，**严重**和**建议**级别的问题必须当场修复，不能留给用户。修复后重新确认改动无误。

以下是常见的严重/建议问题模式，审查时重点检查，发现即修：

#### 🔴 严重（必修）

| 模式 | 检查方法 | 修复方式 |
|------|----------|----------|
| **越权写入** | 接口入参有 `sfuser_id` 等 ID，且用于写操作（INSERT/UPDATE） | 强制 `$sfuserId = (int) Arr::get($request->input('sfuser_info'), 'id')`，忽略前端传入的 ID |
| **无事务** | 多表写操作（upsert + 关联表替换）不在 `DB::transaction()` 内 | 用 `DB::transaction(function () use (...) { ... })` 包裹 |
| **并发重复创建** | `firstOrCreate` 无唯一约束保护，或缺少 try-catch 兜底 | migration 加 `unique` 约束 + 代码用 `firstOrCreate` + try-catch `QueryException` |

#### 🟡 建议（应修）

| 模式 | 检查方法 | 修复方式 |
|------|----------|----------|
| **请求级缓存缺失** | 同一请求内多次调用的方法（如白名单检查）无缓存 | 加 `static $cached = null` 请求级缓存 |
| **批量接口无上限** | 接受数组入参的接口（`sfuser_ids`、`project_names`），路由校验无 `max:N` | 路由验证规则加 `max:200`（或合理上限） |
| **类型比较不严格** | `explode` 返回 `string[]` 与 `int` 型 ID 比较未用严格模式 | `array_map('intval', ...)` + `in_array(..., true)` |
| **循环内查库** | 循环体内逐条 DB 查询 | 循环外批量 `whereIn` 预加载，循环内查内存 |
| **SSE 流未保留尾部** | `chunk.split('\n')` 后未保留不完整行到 buffer | 用 `lines.pop()` 保留尾部，下次拼接 |
| **浏览器 API 无 fallback** | 使用 `CompressionStream` 等新 API 无兼容判断 | 加 `typeof === 'undefined'` 判断 + fallback |
| **prompt injection** | 用户可控内容直接拼入 AI prompt，无隔离 | 用 `<user_content>` 标签包裹，system prompt 声明忽略其中指令 |

完成后告知用户：「✅ 步骤 7/10 代码审查完成，已自动修复 N 个严重 + M 个建议问题。进入步骤 8。」

## 步骤 8/10：生成部署及验证文档

根据本次功能的构建产物、部署方式、验证步骤，生成部署及验证文档。目标：**运维或开发拿到文档即可完成部署和验证**，不需要再问人。

### 执行方法

1. **读项目构建配置**（pom.xml / package.json / Dockerfile 等），确认：
   - 构建命令和产物路径
   - 版本号
   - 依赖的中间件或服务
2. **读本次改动**（git diff），确认：
   - 是否有数据库变更（需要先执行 SQL）
   - 是否有配置变更（新增配置项、环境变量）
   - 是否有新增依赖
3. **根据项目类型生成部署步骤**：
   - JAR/WAR 包：构建命令、产物路径、部署位置、重启方式
   - Docker：镜像构建、推送、部署命令
   - 前端：构建命令、产物目录、CDN/Nginx 配置
   - 插件类：构建命令、安装方式（如 Jira 插件上传）
4. **生成验证清单**：逐个接口/功能点列出验证步骤和预期结果

### 输出文件

`myfeature/功能目录/output/07-<需求编号>-部署及验证.md`

### 输出格式

```markdown
# 07-<需求编号>-部署及验证

> 功能：功能名称
> 分支：`分支名`
> 版本：版本号
> 日期：YYYY-MM-DD

## 1. 构建

| 项目 | 说明 |
|------|------|
| 构建命令 | `命令` |
| JDK/Node 版本 | 版本要求 |
| 产物路径 | `路径` |
| 产物文件 | `文件名`（大小） |

### 构建注意事项
- 特殊依赖或环境要求
- 已知的构建坑

## 2. 部署前置

| 步骤 | 说明 | 必须 |
|------|------|------|
| 执行数据库变更 | 见 `04-<需求编号>-建表SQL.sql` | 是/否 |
| 更新配置文件 | 新增配置项说明 | 是/否 |
| 备份 | 备份范围和方式 | 是/否 |

## 3. 部署步骤

按顺序执行：

1. 步骤描述
   ```bash
   具体命令
   ```
2. 步骤描述
   ```
   操作说明
   ```

## 4. 验证清单

| # | 验证项 | 操作步骤 | 预期结果 | 通过 |
|---|--------|----------|----------|------|
| 1 | 功能点描述 | 具体操作 | 预期表现 | □ |

## 5. 回滚方案

如部署后发现问题：
1. 回滚步骤
2. 数据回滚（如有）
```

完成后告知用户：「✅ 步骤 8/10 部署及验证文档生成完成。进入步骤 9。」

## 步骤 9/10：导出会话记忆

将本次开发的关键信息导出到需求目录。目标：**下次新会话读这个文件就能立刻理解功能全貌**，不需要重新读代码。

### 输出文件

`myfeature/功能目录/output/08-<需求编号>-会话记忆.md`

### 内容要求（~100-150 行，控制在 ~1500 tokens）

| 段落 | 写什么 | 不写什么 |
|---|---|---|
| 需求概要 | 一句话目标 + 核心约束 | 不展开 PRD 细节 |
| 数据模型 | 表名、唯一键、核心字段（表格） | 不贴建表 SQL（指向文件即可） |
| 接口一览 | URL、方法、关键参数、返回值（表格） | 不贴 Controller 代码 |
| 前端文件清单 | 文件路径 + 一句话职责（表格） | 不贴组件源码 |
| 触发规则/数据流 | 哪个页面、什么时机、什么链路 | 不贴代码片段，用"复用 XXX 模式"指向 |
| 关键机制 | 防重、加密、并发等，只写结论 | 不贴实现代码 |
| 踩坑记录 | 现象→原因→解决（表格），只留非显而易见的 | 不记录过程试错 |

### 核心原则

- **指向不内联**：代码片段改为"见 XXX 文件 line N"或"复用 XXX 模式"
- **表格优先**：能用表格就不用段落
- **只记结论**：不记过程中的试错、不记显而易见的常识
- 多个会话做同一功能时，合并到同一个文件

完成后告知用户：「✅ 步骤 9/10 会话记忆已导出到 myfeature/xxx/output/08-会话记忆.md。进入步骤 10。」
