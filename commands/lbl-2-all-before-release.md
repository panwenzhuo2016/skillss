# 需求收尾全流程

需求开发完成后，按顺序执行以下 8 个步骤，每步完成后再进入下一步。中途发现问题直接修复，不要跳步。

## 参数

$ARGUMENTS — 功能分支名、功能目录路径、或功能名称

## 文件编号规则

所有输出文件放在 `myfeature/功能目录/output/` 下，编号从 `03-` 开始，与 `/lbl-1-superpowers-brainstorming`（头脑风暴）生成的 `01-` 文件衔接。

> 历史编号 02 / 07 / 08 已废弃（日志记录文档 / 部署及验证 / 会话记忆三步移除），新需求**不再生成**这三个文件，编号直接跳过。

### 需求编号提取规则

从功能目录名中匹配 `YYMMDD-NNNN-功能名` 格式，取 `NNNN` 部分作为需求编号。例如目录名 `260429-2424-订阅文档更新通知` → 需求编号为 `2424`。

所有输出文件名格式统一为 `编号-<需求编号>-文件名`：

| 编号 | 来源 | 文件名 | 格式说明 |
|------|------|--------|---------|
| 01 | `/lbl-1-superpowers-brainstorming`（头脑风暴） | `01-<需求编号>-design.html`、`01-<需求编号>-plan.html` | HTML（对外展示） |
| 03 | 本流程 步骤 2 | `03-<需求编号>-后端功能点对照.md` | **MD**（自己跟踪进度，改起来快） |
| 04 | 本流程 步骤 3 | `04-<需求编号>-<功能名>-建表SQL.sql` | SQL |
| 05 | 本流程 步骤 4 | `05-<需求编号>-test-api.js` | JS |
| 06 | 本流程 步骤 5 | `06-<需求编号>-<功能名>-前端接口文档.html` | HTML（**给前端工程师看，值得美化**） |
| 09 | 本流程 步骤 8 | `09-<需求编号>-需求对照.html` | HTML（给爹/产品验收，状态卡片+徽章友好） |
| 10 | 本流程 步骤 7 | `10-<需求编号>-性能优化.md` | **MD**（自己用清单跟进；diff 直观） |

> **格式选择原则**：
> - **MD**：自己/Claude 看，重 token 效率、检索、修改速度
> - **HTML**：给别人看（前端/运维/产品/爹），重视觉表达和清单感

> 步骤 6（代码审查）不生成独立编号文件，审查问题直接修复。
> 步骤 7（性能优化）会生成 `10-性能优化.md` 报告，但优化改动直接落到代码里。

### ⚠️ 文件命名强制规则

**每个子 skill 有自己的文件命名规则，本表必须与子 skill 保持一致。** 执行子 skill 时，必须使用子 skill 定义的完整文件名格式，不得简化或省略功能名：

- `/lbl-2-3-gen-sql` 规定文件名为 `04-功能名-建表SQL.sql` → 必须带功能名
- `/lbl-2-5-frontend-api-doc` 规定文件名为 `06-功能名-前端接口文档.html` → 必须带功能名
- `<功能名>` 用中文，如"同步推送开关"、"模板排序"

## 步骤 6/8：代码审查

执行 `/lbl-2-6-review` skill。

### 审查后自动修复

审查完成后，**严重**和**建议**级别的问题必须当场修复，不能留给用户。修复后重新确认改动无误。

以下是常见的严重/建议问题模式，审查时重点检查，发现即修：

#### 🔴 严重（必修）

| 模式 | 检查方法 | 修复方式 |
|------|----------|----------|
| **越权写入** | 接口入参有 `sfuser_id` 等 ID，且用于写操作（INSERT/UPDATE） | 强制 `$sfuserId = (int) Arr::get($request->input('sfuser_info'), 'id')`，忽略前端传入的 ID |
| **无事务** | 多表写操作（upsert + 关联表替换）不在 `DB::transaction()` 内 | 用 `DB::transaction(function () use (...) { ... })` 包裹 |
| **并发重复创建** | `firstOrCreate` 无唯一约束保护，或缺少 try-catch 兜底 | migration 加 `unique` 约束 + 代码用 `firstOrCreate` + try-catch `QueryException` |

#### 🟡 建议（应修）

| 模式 | 检查方法 | 修复方式 |
|------|----------|----------|
| **请求级缓存缺失** | 同一请求内多次调用的方法（如白名单检查）无缓存 | 加 `static $cached = null` 请求级缓存 |
| **批量接口无上限** | 接受数组入参的接口（`sfuser_ids`、`project_names`），路由校验无 `max:N` | 路由验证规则加 `max:200`（或合理上限） |
| **类型比较不严格** | `explode` 返回 `string[]` 与 `int` 型 ID 比较未用严格模式 | `array_map('intval', ...)` + `in_array(..., true)` |
| **循环内查库** | 循环体内逐条 DB 查询 | 循环外批量 `whereIn` 预加载，循环内查内存 |
| **SSE 流未保留尾部** | `chunk.split('\n')` 后未保留不完整行到 buffer | 用 `lines.pop()` 保留尾部，下次拼接 |
| **浏览器 API 无 fallback** | 使用 `CompressionStream` 等新 API 无兼容判断 | 加 `typeof === 'undefined'` 判断 + fallback |
| **prompt injection** | 用户可控内容直接拼入 AI prompt，无隔离 | 用 `<user_content>` 标签包裹，system prompt 声明忽略其中指令 |

完成后告知用户：「✅ 步骤 6/8 代码审查完成，已自动修复 N 个严重 + M 个建议问题。进入步骤 7。」

## 步骤 7/8：性能优化（不改业务逻辑）

执行 `/lbl-2-8-perf-optimize` skill。

扫描本次改动涉及的代码，找出性能热点（循环内多次完整遍历、昂贵操作在过滤前、重复 lookup、N+1 查询、可下推到 SQL/JQL 的过滤条件等），按 **"不改业务逻辑"** 原则进行优化。

### 输出文件

`myfeature/<功能目录>/output/10-<需求编号>-性能优化.md`

### 关键检查点（高收益）

| 模式 | 优化方式 |
|------|----------|
| 循环内多次完整遍历同一集合 | 单次遍历按 key 分桶（batch 提取） |
| 昂贵操作在过滤前 | 重排：先做便宜过滤 |
| 重复 lookup 同一对象 | volatile 缓存类级别共享 |
| N+1 查询 | 循环外批量预加载 |
| 内存过滤可下推到数据层 | 把过滤条件移到 SQL/JQL |

### 强制约束

- 优化前后业务输出必须一致（git diff 仅性能相关）
- 编译通过 `mvn package -DskipTests`（或对应构建命令）
- 关键路径加 `elapsed={}ms` 日志，部署后实测验证

完成后告知用户：「✅ 步骤 7/8 性能优化完成，共优化 N 处。进入步骤 8。」
