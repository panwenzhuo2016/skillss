# 前端接口文档生成

根据后端代码实现，生成「前端接口文档」。前端开发拿到文档即可开始联调，无需再看后端代码。

## 参数

$ARGUMENTS — Controller 代码路径、接口文档路径、或功能名称

## 文档用途

- 前端开发：知道每个接口怎么调、参数怎么传、返回什么结构
- 联调参考：前后端对齐字段名、类型、枚举值
- 测试验证：根据文档构造测试数据

## 执行步骤

### 第一步：收集信息

1. **读 Controller 代码**，提取所有接口：
   - 路径（`@PostResource`/`@GetResource` 的 path）
   - HTTP 方法（GET / POST）
   - 权限注解（`requiredPermission`、`externalOnly`）
   - 参数来源（`@RequestBody` / `@RequestParam` / `@PathVariable`）
2. **读 RO（Request Object）类**，提取每个请求参数：
   - 字段名、类型、校验注解（`@NotNull`/`@NotBlank`/`@Min`/`@Max`/`@Size`）
   - `@Schema` 注解的 description 和 example
   - 默认值（代码中的 `if (xxx == null) xxx = default`）
3. **读 VO（View Object）类**，提取每个响应字段：
   - 字段名、类型
   - `@Schema` / `@JsonFormat` 注解
   - 嵌套对象和列表要递归展开
4. **读 Service 实现代码**，确认业务规则：
   - 状态流转逻辑
   - 互斥/冲突规则（如同时只允许一条生效）
   - 默认值和自动填充字段
   - 权限校验逻辑
5. **读枚举类**，提取所有枚举值及含义
6. **读 CLAUDE.md**，确认项目的 Base URL、认证方式、响应包装结构（如 `ResponseData`）
7. **区分新增接口和修改现有接口**：通过 `git diff` 或对比基准分支，判断每个接口是本次新增的还是已有接口的修改。修改现有接口需明确说明本次变更了什么（如新增字段、改变排序规则等）

### 第二步：组织文档结构

文档由以下部分组成，按顺序排列：

#### 1. 接口概览表
按使用场景分组，每组一个表格，列出接口路径、**类型（新增/修改现有）**、入口（用户在哪个页面触发）、权限要求。

#### 2. 枚举说明
列出所有本次功能用到的枚举，每个枚举一个表格（值 + 说明）。

#### 3. 逐个接口详情
每个接口包含：
- 请求方式和路径
- 请求参数表（参数名 / 类型 / 必填 / 说明）
- 请求示例（完整 JSON）
- 响应示例（完整 JSON，用真实结构和合理的示例值）
- 字段说明表（响应特有字段的类型和含义）
- 业务规则（影响前端逻辑的后端规则，如互斥、默认值、时间限制等）

#### 4. 统一响应结构（如有复用）
多个接口共用同一响应结构时，提取为独立 section，其他接口引用。

#### 5. 页面动作与接口对应
一张总表，列出用户在前端的每个操作对应调用哪个接口，让前端一目了然。

#### 6. 前端展示说明
前端需要知道的交互细节：
- 页面入口和调用时机
- 数据展示规则（排序、优先级、条数限制）
- 特殊交互（弹窗触发条件、红点逻辑、轮播等）
- 注意事项（Long 精度、时区、深色模式等）

### 第三步：生成文档

输出格式（固定模板）：

```markdown
# 06-<需求编号>-功能名-前端接口文档

## 接口概览

### 场景名称（如：管理后台）

| 接口 | 类型 | 入口 | 权限 |
|------|------|------|------|
| `METHOD /api/v1/xxx` | **新增** / **修改现有** | 页面 > 操作 | 角色说明 |

---

## 枚举说明

### enumName 枚举中文名
| 值 | 说明 |
|---|------|
| 0 | 含义 |

---

## 接口 1：接口名称【新增接口】或【修改现有接口】

> 若为修改现有接口，此处必须写明本次变更内容，如：响应新增 `xxx` 字段，返回列表改为按 `xxx` 排序。请求参数无变化。

### 请求

```
METHOD /api/v1/xxx
```

**Request Body (JSON)：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| fieldName | string | 是 | 字段说明（含默认值、格式要求） |

**请求示例：**

```json
{
  "fieldName": "示例值"
}
```

**响应：**

```json
{
  "success": true,
  "code": 200,
  "data": { ... }
}
```

**业务规则：**
- 规则 1
- 规则 2

---

## 统一响应结构名称

```json
{ ... }
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| fieldName | string | 含义 |

---

## 页面动作与接口对应

| 页面动作 | 调用接口 |
|----------|----------|
| 进入 XX 页面，加载列表 | `GET /api/v1/xxx` |
| 点击「创建」按钮 | `POST /api/v1/xxx` |
| 点击「删除」按钮 | `POST /api/v1/xxx/delete` |

---

## 前端展示说明

### 1. 场景名称

- 调用时机和接口
- 展示规则
- 交互细节
```

### 第四步：检查完整性

逐项检查：

1. **每个 Controller 接口都有文档**：对照 Controller 代码，不遗漏
2. **参数和响应字段完整**：对照 RO/VO 类，每个字段都列出
3. **必填标注准确**：对照 `@NotNull`/`@NotBlank` 等注解
4. **枚举值完整**：对照枚举类，每个值都列出
5. **业务规则覆盖**：Service 中影响前端的逻辑都写入了"业务规则"
6. **示例值合理**：JSON 示例用真实的数据结构和合理的值，不是 `xxx`
7. **类型准确**：特别注意 Long 类型（前端精度问题）、时间格式、null 的含义
8. **前端注意事项**：Long 精度、时区（Asia/Shanghai）、深色模式、响应式等

### 注意事项

- 文件名格式：`06-<需求编号>-功能名-前端接口文档.md`（需求编号从功能目录名中匹配 `YYMMDD-NNNN-功能名` 格式取 `NNNN`，功能名用中文，如 `06-2424-模板排序-前端接口文档.md`），放在对应的 `myfeature/功能目录/output/` 目录下
- 路径统一写完整路径（含 `/api/v1` 前缀），与前端实际调用一致
- 参数说明要写**前端怎么传**，不是后端怎么存。例如：`publishTime` 说明传"当前时间"还是"未来时间"
- 响应字段说明要写**前端怎么用**。例如：`hasRead=false` 时展示弹窗
- 如果有 Query 参数和 Body 参数混用的接口，分别列出
- 同一响应结构被 3 个以上接口复用时，提取为统一结构，避免重复
- 业务规则只写影响前端逻辑的（如互斥、默认值），后端内部实现细节不写
- 如果接口有分页，明确写出分页参数名（`pageNo`/`page`）、默认值、最大值
