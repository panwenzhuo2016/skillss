# 需求分析

分析需求并拆解为可执行的开发任务。

## 参数

$ARGUMENTS — 需求描述

## 执行步骤

1. 理解需求描述
2. 分析涉及的模块和改动范围
3. 拆解为具体的开发任务
4. 识别风险和依赖

## 输出内容

### 需求理解
- 用一段话复述需求，确认理解正确
- 列出需要确认的疑问点

### 技术分析
- 涉及哪些模块/服务
- 需要修改哪些文件（大致范围）
- 是否需要数据库变更
- 是否需要新增 API 接口
- 是否影响现有功能

### 任务拆解

```
1. [后端] 数据库表结构设计
   - 新增 xxx 表
   - xxx 表新增字段

2. [后端] API 接口开发
   - POST /api/v1/xxx — 创建
   - GET /api/v1/xxx — 查询列表

3. [前端] 页面开发
   - 列表页面
   - 表单页面

4. [测试] 测试用例
   - 单元测试
   - 接口测试
```

### 风险识别
- 技术风险（不确定的技术点）
- 依赖风险（依赖其他人/系统）
- 数据风险（数据迁移、兼容性）

## 规则

- 任务拆解粒度要合适（每个任务 0.5-2 天的工作量）
- 标注任务间的依赖关系
- 不做时间估算（只拆解任务）
