# Bug 修复全流程

从 Bug 描述出发，完成定位、分析、修复、验证的完整闭环。

## 参数

$ARGUMENTS — Bug 描述、错误信息、Sentry 链接、截图路径、或 Issue 编号

## Workflow

在动手之前，先列出完整操作清单，**等用户确认后再执行**。

```text
Bug 修复进度：
- [ ] Step 0: 收集信息，明确问题
- [ ] Step 1: 定位问题代码
- [ ] Step 2: 分析根因，列出假设
- [ ] Step 3: 验证假设，确认根因
- [ ] Step 4: 制定修复方案
- [ ] Step 5: 实施修复
- [ ] Step 6: 验证修复
- [ ] Step 7: 总结输出
```

### Step 0: 收集信息，明确问题

1. **不要假设项目**，根据当前工作目录判断：
   - 如果当前目录在某个 git 仓库内，确认是否就是 bug 所在的仓库
   - 如果不确定，**问用户**："哪个项目？"
2. **解析用户输入**，提取关键信息：

   | 用户给了什么 | 提取什么 |
   |-------------|---------|
   | 错误堆栈 | 异常类型、文件名、行号、调用链 |
   | Sentry 链接 | 执行 `/sentry-investigate` 获取详情 |
   | 截图 | 用 Read 工具查看截图，识别错误信息 |
   | Issue 编号 | 用 `gh issue view` 或 GitLab API 获取描述 |
   | 口头描述 | 提取关键词：哪个功能、什么现象、什么条件下触发 |

3. **补全缺失信息**，如果以下信息缺失，**问用户**：
   - 环境：测试环境还是生产环境？
   - 复现：必现还是偶现？复现步骤是什么？
   - 时间：什么时候开始出现的？最近有没有发版？
4. **确认后记录**：
   - 项目名和仓库路径
   - Bug 现象一句话描述
   - 环境和复现条件

### Step 1: 定位问题代码

根据 Step 0 收集到的信息，选择最高效的定位路径：

| 有什么线索 | 定位方法 |
|-----------|---------|
| 有堆栈/行号 | 直接 Read 对应文件和行号 |
| 有 Sentry 事件 | 从 Sentry 堆栈提取文件和行号 |
| 有接口路径（如 `/api/user/list`） | Grep 搜索路由定义 → Controller → Service → Mapper |
| 有错误消息文本 | Grep 搜索错误消息字符串，找到抛出位置 |
| 有前端报错 | Grep 搜索组件名/请求路径，找到前端代码，再追到后端接口 |
| 只有功能描述 | Grep 搜索功能关键词，找到相关模块，逐层排查 |

定位过程中顺藤摸瓜，读清完整调用链：
```
入口（Controller/页面） → 业务逻辑（Service） → 数据层（Mapper/SQL） → 配置/拦截器
```

**输出**：列出所有相关文件和行号，标注哪些是重点嫌疑。

### Step 2: 分析根因，列出假设

**必须列出至少 2 个可能的原因**（只有 1 个假设时不允许直接动手）。

按可能性从高到低排列：

```markdown
### 可能原因分析

| # | 假设 | 可能性 | 依据 |
|---|------|--------|------|
| 1 | xxx | 高 | 因为 xxx |
| 2 | xxx | 中 | 因为 xxx |
| 3 | xxx | 低 | 因为 xxx |
```

重点检查这些常见根因：

#### 后端常见
- **NPE**：哪个对象可能为 null？数据来源是什么？
- **SQL 异常**：字段类型不匹配？缺少索引？SQL 拼接错误？
- **并发问题**：有没有竞态条件？事务隔离级别对不对？
- **数据问题**：脏数据？枚举值不在预期范围？
- **配置问题**：环境变量缺失？配置文件差异？
- **权限问题**：拦截器/注解是否正确？多租户隔离是否生效？

#### 前端常见
- **TypeError**：API 返回结构变了？可选链没加？
- **状态异常**：响应式数据没触发更新？竞态条件？
- **白屏**：组件渲染报错被吞了？路由配置错误？
- **请求失败**：CORS？参数格式？后端接口变更？

### Step 3: 验证假设，确认根因

对每个假设逐一验证，**用证据说话**：

| 验证手段 | 适用场景 |
|----------|---------|
| 读代码逻辑 | 确认代码是否真的存在假设中的问题 |
| 查数据库 | 确认数据状态是否符合预期（用 MCP 查） |
| 看 git log | 确认最近有没有相关改动引入 bug |
| 看配置文件 | 确认配置值是否正确 |
| 对比环境 | 测试环境 vs 生产环境的差异 |

```markdown
### 验证结果

| # | 假设 | 验证方法 | 结果 |
|---|------|----------|------|
| 1 | xxx | 读了 XxxService.java:85 | ✅ 确认是根因：xxx |
| 2 | xxx | 查了数据库 xxx 表 | ❌ 排除：数据正常 |
```

**确认根因后，向用户说明**：
- 根因是什么（一句话）
- 为什么会触发（因果链）
- 影响范围（哪些功能/用户受影响）

### Step 4: 制定修复方案

1. **只修 bug 本身**，不做额外重构或优化
2. **如果有多种修复方式，列出对比让用户选**：

   | 方案 | 改动范围 | 风险 | 推荐 |
   |------|---------|------|------|
   | A: xxx | 改 1 个文件 | 低 | ✅ |
   | B: xxx | 改 3 个文件 | 中 | |

3. **列出具体改动点**：
   ```
   修复清单：
   - [ ] 文件A:行号 — 改动描述
   - [ ] 文件B:行号 — 改动描述
   ```
4. **等待用户确认后再动手**

### Step 5: 实施修复

1. 按 Step 4 确认的方案逐个修改
2. 修改过程中检查（对应 CLAUDE.md 规则 13）：
   - **安全**：有无注入/越权？
   - **性能**：有无 N+1/全表扫描？
   - **边界**：null/空集合/零值/并发是否处理？
   - **错误**：异常是否捕获并有明确后续？
3. 在修复点添加关键日志：
   - 按 `/lbl-02-add-logs` 的规范添加日志，但功能名后面追加 `-fixbug` 标识
   - 格式：`[lbl][功能名-fixbug][模块名][操作] 描述, 关键参数`
   - 后端示例：`log.info("[lbl][2087-AI日报-fixbug][UserService][UPDATE] 修复空值判断, userId={}", userId)`
   - 前端示例：`console.warn('[lbl][2087-AI日报-fixbug][UserList][QUERY] 修复 undefined 处理', { id })`
   - 功能名从分支名或用户输入中获取，如果不确定，**问用户**
   - 其余日志级别、参数格式等规范与 `/lbl-02-add-logs` 完全一致
4. 修改范围最小化，**不要顺手改周边代码**

### Step 6: 验证修复

1. **代码层面验证**：
   - 重新读修改后的代码，确认逻辑正确
   - 检查是否引入新问题（回归）
   - 如果有测试，运行测试：
     ```bash
     # Java 项目
     mvn test -pl <module> -Dtest=<TestClass>
     # 前端项目
     npm test -- --filter <test-file>
     ```
2. **数据层面验证**（如涉及数据库改动）：
   - 用 MCP 查询确认 SQL 正确
   - 检查索引是否命中
3. **如果无法本地验证**，告知用户需要在哪个环境验证，以及验证步骤

### Step 7: 总结输出

#### 8.1 控制台输出修复总结

```markdown
## Bug 修复完成

| 项目 | 状态 |
|------|------|
| 项目 | <项目名> |
| Bug 现象 | <一句话> |
| 根因 | <一句话> |
| 修复方案 | <一句话> |
| 改动文件 | N 个文件 |
| 影响范围 | <涉及的功能/模块> |
```

如果 bug 涉及的模式可能在其他地方也存在，主动 Grep 搜索，确认是否需要批量修复。

#### 8.2 生成修复记录文档

文件格式：

```markdown

- **影响范围**：<哪些功能/用户受影响>
```

## Quality Rules

- **先收集再动手**：Step 0-1 收集信息和定位代码，必须在 Step 5 之前完成，不要跳步。
- **至少 2 个假设**：只有 1 个假设时不允许直接修复，必须再想一个可能性或向用户确认。
- **用证据说话**：每个假设必须有验证方法和结论，不靠猜。
- **最小化改动**：只改导致 bug 的代码，不修改周边代码，不顺手重构。
- **关联排查**：修完后搜索同类模式，确认是否有其他隐患。
- **确认再动手**：Step 4 列出改动点后，等用户确认再执行 Step 5。
