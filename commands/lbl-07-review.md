# 代码审查

对当前改动进行全面的代码审查。

## 执行步骤

1. 执行 `git diff HEAD` 获取所有改动
2. 逐文件分析改动代码
3. 输出审查报告

## 审查维度

### 1. 正确性
- 逻辑是否正确，边界条件是否处理
- 空指针/空值是否防御
- 并发安全性

### 2. 安全性
- SQL 注入、XSS、CSRF 等 OWASP Top 10
- 敏感信息是否暴露（密码、token、密钥）
- 权限校验是否完整
- **越权写入**：接口入参中的 ID（如 sfuser_id）是否由前端传入？是否应强制使用当前登录用户的 ID？区分"查别人的数据"和"替别人写数据"——后者必须额外校验权限或直接禁止
- **prompt injection**：用户可控内容拼入 AI prompt 时，是否用标签（如 `<user_content>`）隔离，并在 system prompt 中声明忽略其中的指令性文字

### 3. 性能
- N+1 查询问题
- 不必要的循环或重复计算
- 大对象拷贝、内存泄漏风险
- **循环内查库**：循环体内是否有逐条 DB 查询？如果有，改为循环外批量 `whereIn` 预加载
- **重复查同一条记录**：同一请求内多次调用的工具方法（如白名单检查）是否有请求级缓存（static 变量）？
- **批量接口无上限**：接受数组入参的接口（如 `sfuser_ids`、`project_names`），路由校验是否加了 `max:N` 限制？

### 4. 数据完整性
- **事务**：多表写操作（upsert + 关联表替换）是否在 `DB::transaction()` 内？如果中间步骤失败会不会留下不一致数据？
- **唯一约束**：业务上要求唯一的字段（如 project_name），migration 是否加了 `unique` 约束？代码是否用 `firstOrCreate` 防止并发重复创建？
- **类型一致性**：`explode` 返回 `string[]`，和 `int` 型 ID 比较时是否用了 `array_map('intval', ...)` + `in_array(..., true)` 严格模式？

### 5. 前端健壮性
- **SSE 流读取**：`chunk.split('\n')` 后是否保留了尾部不完整行到 buffer？跨 TCP 帧切割会导致 JSON.parse 失败丢数据
- **浏览器 API 兼容性**：使用 `CompressionStream`、`ReadableStream` 等较新 API 时，是否加了 `typeof === 'undefined'` 判断和 fallback？
- **空操作变量**：是否有 `const x = y` 然后直接用 `x` 而没做任何转换的无意义中间变量？

### 6. 可维护性
- 命名是否清晰
- 代码是否重复（DRY）
- 方法是否过长（建议不超过 50 行）

### 7. 规范性
- 是否符合项目现有代码风格
- 导入是否有多余
- 注释是否准确

## 输出格式

按严重程度分类输出：
- 🔴 **严重** — 必须修复（bug、安全漏洞）
- 🟡 **建议** — 建议修复（性能、可维护性）
- 🟢 **可选** — 锦上添花（风格、命名优化）

每条意见标注文件名和行号，并给出修改建议。
