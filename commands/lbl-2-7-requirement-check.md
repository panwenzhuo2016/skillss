# 需求细节对照检查

逐条核对需求文档中的每一项功能点、交互细节、边界规则，与后端代码实现一一比对，确保无遗漏、无偏差。

## 参数

$ARGUMENTS — 需求文档路径，或功能分支名/功能目录路径（自动从 myfeature 目录查找需求文档）

## 执行方法

### 1. 定位需求文档

按优先级查找：
1. 参数直接指定的 `.md` 文件
2. `myfeature/<功能目录>/` 下的需求文档（排除 `output/` 子目录中的生成文件）
3. 如果找不到，提示用户指定路径

### 2. 定位代码

1. 从当前 git 分支确认是否在功能分支上
2. 如果不在，搜索匹配的功能分支并切换（切换前 `git stash`）
3. 找到功能相关的所有代码文件（Controller / Manager / Service / Entity / RO / VO / Consumer / Publisher / Config 等）

### 3. 逐条核对

**逐行阅读需求文档**，将每一个功能点、交互细节、边界规则、异常处理拆解为独立检查项，然后对每个检查项：

1. **找到对应代码** — 精确定位到文件名和行号
2. **验证实现是否正确** — 不只是"有没有"，还要验证逻辑细节（如阈值、文案、时间单位等）
3. **标记状态** — ✅ 已实现 / ⚠️ 部分实现或待定 / ❌ 未实现

### 4. 隐含需求检查

需求文档未明确写出但代码应处理的场景：

| 类别 | 检查项 |
|------|--------|
| 安全 | 操作者权限校验、用户数据隔离、SQL 注入防护 |
| 幂等 | 重复操作不报错、不产生脏数据 |
| 边界 | null/空列表/不存在的 ID/已删除的数据 |
| 性能 | N+1 查询、全表扫描、大数据量内存加载 |
| 事务 | 多表写操作是否在事务内、事务范围是否最小化 |
| 异步 | MQ 消息发送时机（事务提交后）、消费失败处理 |
| 推送 | WebSocket/IM 推送是否覆盖所有渠道 |

### 5. 文案/格式精确匹配

需求中出现的**精确文案**必须与代码中的字符串逐字对比：
- IM 通知文案格式
- 错误提示文案
- 聚合消息文案
- URL 拼接格式

## 输出格式

按需求文档的章节结构组织，每个章节一个表格：

```markdown
## 需求 vs 后端代码逐条核对

### 一、章节名

| # | 需求 | 后端实现 | 状态 |
|---|------|----------|------|
| 1.1 | 需求描述 | 代码实现说明（文件名:行号） | ✅/⚠️/❌ |

### 隐含需求

| # | 检查点 | 后端实现 | 状态 |
|---|--------|----------|------|
| H.1 | 检查项 | 实现说明 | ✅/⚠️/❌ |

---

### 核对结论

**N 个检查点中：X 个 ✅，Y 个 ⚠️，Z 个 ❌**

⚠️ 待定项：
- 描述 + 原因

❌ 未实现项：
- 描述 + 影响范围 + 建议
```

## 发现问题的处理

- **❌ 未实现**：明确告知用户，列出影响范围，问是否需要补充实现
- **⚠️ 部分实现**：说明差异，问是否需要修正（如链接待前端路由确定，则标注"待前端"）
- **文案不匹配**：直接修复代码中的文案，使其与需求一致
- **逻辑偏差**：描述需求要求 vs 实际实现的差异，让用户决策

## 注意事项

- **不要凭记忆**，必须同时打开需求文档和代码文件逐行对比
- **精确到字符**：文案中的标点符号、空格、书名号都要对
- **区分前后端职责**：前端负责的交互（如 toast、弹窗 UI）标注"前端负责"即可，不算未实现
- **数值精确匹配**：阈值（如 5 分钟、9 条、第 10 条）必须与代码中的常量值一致
- **输出格式**：增强 HTML（纯 HTML+CSS，不依赖外部库），使用表格样式（斑马纹、hover 高亮）、状态徽章（✅/⚠️/❌）、可折叠章节分组、侧边栏导航、统计仪表盘（通过/待定/未实现数量）
