# LLM 服务端推理：从收到请求到返回 Token 的 10 步

> 基于 Anthropic 公开技术博客、论文、以及 LLM Serving 通用架构推断
> 无法直接验证，属于合理推断，标注 [推断] 的部分不保证准确

## 1. 请求接入 & 路由调度
API Gateway 收到请求后：
- 鉴权（API Key 校验、配额检查、限流判断）
- 根据 `model` 字段路由到对应的模型集群（Opus/Sonnet/Haiku 各自独立集群）
- 负载均衡：选择一个空闲的推理节点（GPU 实例）
- 如果所有节点都满载 → 返回 529 Overloaded

```
Request → API Gateway → Auth/Rate Limit → Router → GPU Cluster
```

## 2. Tokenization（分词）
把请求中的所有文本转成 token ID 序列：
- system prompt → token IDs
- messages（所有历史对话）→ token IDs
- tools 定义 → token IDs
- 使用 Anthropic 自己的 tokenizer（类 BPE 算法）
- 计算总 input_tokens，校验是否超过模型 context_length 上限（Opus 约 200K）

## 3. Prompt Cache 查找 [KV Cache]
这是 Anthropic 的 **Prompt Caching** 机制，核心是复用 KV Cache：
- 从请求头部开始，逐 token 比对是否与缓存中的 prefix 匹配
- **命中**：直接加载已缓存的 KV Cache（Key-Value 矩阵），跳过这些 token 的 Prefill 计算
  - 报告 `cache_read_input_tokens`（省钱省时间）
- **未命中**：标记需要完整计算，计算完后写入缓存
  - 报告 `cache_creation_input_tokens`（首次会慢一点）
- 缓存粒度：按 token 前缀匹配，system + tools 通常是稳定前缀，命中率最高

```
[system: 5000 tokens][tools: 3000 tokens][messages: 4000 tokens]
 ←── 这段通常缓存命中 ──→  ←── 这段每次不同 ──→
```

## 4. Prefill 阶段（并行计算所有 input tokens）
Transformer 推理的第一阶段，也叫 **Prompt Processing**：
- 把所有未命中缓存的 input tokens **并行** 送入模型
- 逐层计算 Transformer：
  - 每层：Multi-Head Attention → LayerNorm → FFN → LayerNorm
  - Claude Opus 预估 100+ 层，数千亿参数 [推断]
- 计算过程中生成每一层的 **KV Cache**（Key 和 Value 矩阵），存入显存
- 这一步是 **计算密集型**（Compute Bound），所有 input tokens 一起算，GPU 利用率最高
- 耗时与 input_tokens 数量成正比 —— 你的 system prompt 越长，这步越慢

```
Input tokens → [Layer 1] → [Layer 2] → ... → [Layer N] → KV Cache 写入显存
              ←────── 所有 token 并行计算，GPU 满载 ──────→
```

## 5. Extended Thinking（内部思考链）
如果开启了 thinking（Claude Code 默认开启）：
- 模型开始**自回归生成** thinking tokens
- 每个 thinking token 的生成流程同第 7 步的 Decode
- thinking 内容对用户不可见（或部分可见），但消耗 `budget_tokens`
- 模型在 thinking 中进行：推理、规划、自我纠错、方案对比
- 生成特殊的结束标记时停止 thinking，转入正式输出
- [推断] thinking tokens 也会写入 KV Cache，影响后续生成的注意力

```
Prefill 完成 → 开始生成 thinking tokens（一个接一个）
→ "让我分析一下这个问题..."
→ "方案A: ... 方案B: ..."
→ "选择方案A因为..."
→ [thinking 结束标记] → 转入正式输出
```

## 6. Sampling 策略准备
正式输出前，设置采样参数：
- `temperature`：控制随机性（Claude Code 默认 1.0）
- `top_k` / `top_p`：截断低概率 token
- 对于 tool_use 输出，[推断] 可能用更低 temperature 保证 JSON 格式正确
- 构建 logit processor 链（重复惩罚、停止序列检测等）

## 7. Decode 阶段（逐 token 自回归生成）
Transformer 推理的第二阶段，一个 token 一个 token 地生成：

**单个 token 的生成过程：**
1. 取上一个 token 的 embedding
2. 逐层计算 Attention：当前 token 的 Query 与所有历史 token 的 KV Cache 做注意力计算
3. 经过 FFN 层得到 logits（词表大小的向量，每个位置代表一个 token 的得分）
4. 应用 temperature → softmax → top_k/top_p 截断 → 采样出一个 token
5. 把新 token 的 KV 写入 Cache，供下一个 token 使用
6. 重复，直到生成 stop token 或达到 max_tokens

- 这一步是 **显存带宽密集型**（Memory Bound），每生成一个 token 都要读取整个 KV Cache
- 速度：约 30-80 tokens/sec（取决于模型大小和 batch size）

```
token[0] → Attention(Q·K^T) → FFN → logits → sample → "好"
                                                         ↓
token[1] → Attention(Q·K^T) → FFN → logits → sample → "的"
                                                         ↓
token[2] → Attention(Q·K^T) → FFN → logits → sample → ","
... 逐个生成，每个都依赖前面所有 token 的 KV Cache
```

## 8. 结构化输出控制（Text vs Tool Use）
模型生成过程中需要遵守输出格式约束：
- **文本输出**：自由生成，流式推送每个 token
- **工具调用**：生成符合 JSON Schema 的结构化输出
  - [推断] 使用 Constrained Decoding（受限解码）：在 logits 上 mask 掉不合法的 token
  - 例如：生成 `{"name": "Edit"` 后，下一个 token 必须是 `,` 或 `}`，其他 token 的概率被置零
  - 保证生成的 JSON 始终合法，参数名和类型符合 tool schema
- `tool_choice` 参数影响：`auto` 让模型自己决定，`any` 强制调工具，`tool` 指定调哪个

## 9. Streaming SSE 推送
每生成一个（或一小批）token，立即通过 SSE 推送给客户端：

```
# 第一个 content block 开始
event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

# 逐 token 推送
event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"好的"}}

# 如果是工具调用，参数增量推送
event: content_block_delta
data: {"type":"content_block_delta","index":1,"delta":{"type":"input_json_delta","partial_json":"\"file_pa"}}
```

推送与生成是**流水线并行**的：模型生成 token N+1 的同时，token N 已经在推送给客户端了。

## 10. 停止判断 & 资源回收
模型停止生成的条件（按优先级）：
1. 生成了 `end_turn` 特殊 token → `stop_reason: "end_turn"`
2. 生成了工具调用的完整 JSON → `stop_reason: "tool_use"` [推断]
3. 达到 `max_tokens` 限制 → `stop_reason: "max_tokens"`
4. 命中 `stop_sequences` → `stop_reason: "stop_sequence"`

停止后：
- 统计 usage（input_tokens、output_tokens、cache_read、cache_creation）
- 发送 `message_stop` 事件
- [推断] KV Cache 根据策略决定保留（供后续对话复用）或释放显存
- GPU 资源归还调度器，准备接收下一个请求

## 流程图

```
请求到达 API Gateway
  ↓
鉴权/限流/路由 → 选择 GPU 节点
  ↓
Tokenization（文本 → token IDs）
  ↓
Prompt Cache 查找
  ├─ 命中 → 加载 KV Cache（跳过已缓存部分的计算）
  └─ 未命中 → 标记需完整计算
  ↓
Prefill（未缓存的 input tokens 并行计算 → 生成 KV Cache）
  ↓
Extended Thinking（自回归生成思考链，消耗 budget_tokens）
  ↓
Sampling 策略准备（temperature / top_k / top_p）
  ↓
Decode（逐 token 生成）
  ├─ Attention(Q × KV Cache) → FFN → logits → sample
  ├─ 结构化约束（工具调用时 JSON Schema 受限解码）
  └─ 每个 token 立即 SSE 推送
  ↓
停止条件触发（end_turn / tool_use / max_tokens）
  ↓
Usage 统计 → message_stop → KV Cache 策略 → 资源回收
```

## 性能瓶颈在哪

| 阶段 | 瓶颈类型 | 影响 |
|------|----------|------|
| Prefill | **计算密集** (Compute Bound) | input 越长越慢，但 GPU 利用率高 |
| Decode | **显存带宽密集** (Memory Bound) | 每个 token 都要读全部 KV Cache，GPU 利用率低 |
| Thinking | 同 Decode | thinking 越长，正式输出等待越久 |
| Cache Miss | 额外计算 | 首次请求或 prompt 变化大时，Prefill 更慢 |

**你感受到的"等待"主要来自：**
- Prefill（处理你的长 system prompt + 对话历史）
- Thinking（模型内部思考）
- 队列等待（高峰期排队等 GPU）

**你感受到的"打字速度"就是 Decode 速度：**
- 约 30-80 tokens/sec，取决于模型大小和并发负载

## 10 步深究价值排名（从最值得到最不值得）

| 排名 | 步骤 | 理由 |
|------|------|------|
| **1** | **7. Decode（逐token自回归生成）** | LLM 推理的核心瓶颈。Attention 怎么算、KV Cache 怎么管理、FlashAttention/PagedAttention 怎么优化、Speculative Decoding 怎么加速——推理优化领域 80% 的论文都在搞这一步 |
| **2** | **4. Prefill（并行计算input tokens）** | 决定 TTFT 的关键。长上下文（200K）下 Prefill 本身就要好几秒。怎么分块、怎么调度 Prefill 和 Decode 不互相抢 GPU——Serving 系统设计的核心难题 |
| **3** | **8. 结构化输出控制（Constrained Decoding）** | 让 LLM 从"随便说话"变成"可编程工具"的关键技术。怎么在不破坏生成质量的前提下强制输出合法 JSON——直接决定 Agent/Function Calling 的可靠性 |
| **4** | **5. Extended Thinking（思考链）** | Claude 和 o1/o3 系列的核心差异化能力。怎么训练模型"会思考"、thinking 预算怎么分配、思考链和输出质量的关系——当前 AI 能力天花板的突破口 |
| **5** | **3. Prompt Cache（KV Cache复用）** | 直接决定成本和延迟。缓存策略、淘汰机制、多用户间能否共享前缀缓存——LLM 商业化运营的关键技术 |
| **6** | **6. Sampling 策略** | temperature/top_k/top_p 看着简单，但直接影响输出质量。什么时候该确定性、什么时候该随机、怎么平衡创造力和准确性——调参的本质就在这 |
| **7** | **1. 请求路由调度** | 大规模 GPU 集群怎么调度、负载均衡、异构硬件处理——infra 层面的硬功夫，但和 LLM 本身关系不大 |
| **8** | **9. Streaming SSE 推送** | 工程实现，流水线并行，没太多深度，但做好了体验差别很大 |
| **9** | **2. Tokenization** | BPE 算法已经很成熟，除非自己训模型设计 tokenizer，否则了解原理就够 |
| **10** | **10. 停止判断 & 资源回收** | 纯工程逻辑，条件判断 + 资源释放，没什么可深究的 |

**如果只挑一个深究**：第 7 步 Decode。因为这一步藏着整个 LLM 推理的核心矛盾——每生成一个 token 都要读一遍完整 KV Cache，GPU 算力利用率只有 1-5%，剩下全在等显存搬数据。所有优化技术（FlashAttention、PagedAttention、GQA/MQA、Speculative Decoding、Continuous Batching）本质上都在解决这一个问题。
