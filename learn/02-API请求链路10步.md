# messages.create 请求全链路：从发出到返回的 10 步

> 基于 cli.js (v2.1.78) 逆向分析 + Anthropic API 文档

## 1. Token 预算计算
发请求前，先估算当前 messages 的 token 数（86 处 `tokenize`），算出还剩多少空间给 `max_tokens`（输出上限）和 `budget_tokens`（thinking 预算）。超了就先触发压缩/截断。

## 2. 组装请求体
拼一个 JSON 请求：
```json
{
  "model": "claude-opus-4-6",
  "max_tokens": 16384,
  "system": [{ "text": "...", "cache_control": {"type": "ephemeral"} }],
  "messages": [...],
  "tools": [Read, Edit, Write, Bash, Glob, Grep, ...共30+个],
  "thinking": { "type": "enabled", "budget_tokens": ... },
  "stream": true
}
```
关键点：`system` 和 `tools` 会打上 `cache_control: ephemeral` 标记（34 处），让服务端走 **Prompt Caching**，避免重复计算。

## 3. 发起 HTTPS 请求
SDK 向 `ANTHROPIC_BASE_URL`（如 `tower-ai.yottastudios.com/zi/proxy`）发起 POST 请求，带上 `ANTHROPIC_API_KEY`，开启 SSE 流式连接。

## 4. 服务端：Prompt Cache 命中检查
服务端收到请求，先检查 system prompt + tools 定义有没有命中缓存（`cache_read_input_tokens`）。命中了就省掉这些 token 的处理费用和时间，没命中就创建新缓存（`cache_creation_input_tokens`）。

## 5. 服务端：模型推理（Thinking）
模型开始推理。如果开了 extended thinking（361 处引用），模型先在内部"思考"，消耗 `budget_tokens` 的预算。这一步是 **TTFT（Time To First Token）** 的主要来源 —— 等的那几秒就在这。

## 6. 服务端：生成输出（Streaming）
模型逐 token 生成输出，通过 SSE 流式推送。每个 event 格式：
```
event: content_block_delta
data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"好"}}
```
文本和工具调用是分开的 content_block，工具参数通过 `input_json_delta` 增量传输。

## 7. 客户端：流式解析 & 渲染
CLI 收到 SSE 事件流，逐个解析：
- `message_start` → 记录 message id、model 等元信息
- `content_block_start` → 判断是 text 还是 tool_use
- `content_block_delta` → 文本实时渲染 / 工具参数拼接
- `content_block_stop` → 一个块结束

## 8. 客户端：错误处理 & 重试
如果中途出问题：
- **429 Rate Limit**（246 处引用）→ 指数退避重试（68 处 backoff）
- **529 Overloaded**（76 处引用）→ 同上，等服务端恢复
- **网络断开** → 重试机制兜底（590 处 retry 相关代码）

## 9. 客户端：接收完成 & Usage 统计
收到 `message_stop`，提取 usage 信息：
```json
{
  "input_tokens": 12000,
  "output_tokens": 800,
  "cache_read_input_tokens": 10000,
  "cache_creation_input_tokens": 0
}
```
用于终端底部的 token 消耗统计显示。

## 10. 解析最终结果 & 决定下一步
把完整 response 解析成结构化数据：
- `content` 数组里的每个 block 分类处理
- `stop_reason` 决定命运：
  - `"end_turn"` → 模型说完了，结束 mainLoop
  - `"tool_use"` → 模型要调工具，进入权限检查→执行→回填→再次调 API
  - `"max_tokens"` → 输出被截断，可能需要继续

## 流程图

```
Token预算计算 → 组装请求体(system+messages+tools)
                         ↓
              发起HTTPS POST (SSE streaming)
                         ↓
           ┌─────── 服务端处理 ───────┐
           │ Cache命中检查             │
           │     ↓                    │
           │ 模型Thinking(TTFT等待)    │
           │     ↓                    │
           │ 逐token生成 → SSE推送     │
           └──────────────────────────┘
                         ↓
              客户端流式解析 & 渲染
                         ↓
                 错误? → 退避重试
                         ↓
              接收完成 → Usage统计
                         ↓
              stop_reason判断下一步
              ├─ end_turn → 结束
              ├─ tool_use → 执行工具 → 再次请求
              └─ max_tokens → 继续生成
```
