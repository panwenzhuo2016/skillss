# 停止判断 & 资源回收：从生成结束到释放的 10 步

> 对应服务端推理流程第 10 步的深度拆解

## 1. 停止条件检测
每生成一个 token，检查是否满足停止条件（按优先级）：

| 优先级 | 条件 | stop_reason | 说明 |
|--------|------|------------|------|
| 1 | 达到 `max_tokens` | `"max_tokens"` | 硬上限，强制截断 |
| 2 | 生成 EOS token | `"end_turn"` | 模型自行决定说完了 |
| 3 | 工具调用 JSON 完成 | `"tool_use"` | 完整的 tool_use block 闭合 |
| 4 | 命中 `stop_sequences` | `"stop_sequence"` | 生成了用户指定的停止文本 |

```
每个 token 生成后：
  if output_count >= max_tokens → STOP (max_tokens)
  if token == EOS_TOKEN        → STOP (end_turn)
  if tool_json_complete()      → STOP (tool_use)
  if text.endswith(stop_seq)   → STOP (stop_sequence)
  else → CONTINUE
```

## 2. 最终 Token 处理
停止条件触发后的最后一个 token：
- `stop_sequence` 停止：最后一个 token（或触发序列）**不包含**在输出中
- `max_tokens` 停止：最后一个 token 正常输出，但可能截断到一半
- `end_turn` 停止：EOS token 不输出给用户
- `tool_use` 停止：确保 JSON 的最后 `}` 已推送

## 3. Usage 统计汇总
计算本次请求的完整用量：
```json
{
  "input_tokens": 12000,              // Tokenization 阶段算的
  "output_tokens": 800,               // Decode 实际生成的 token 数
  "cache_read_input_tokens": 8000,    // Prompt Cache 命中的
  "cache_creation_input_tokens": 4000 // 新建缓存的
}
```

- output_tokens 包含 thinking tokens（如果有）
- 费用 = input 费用 + output 费用 + cache 费用（各有不同单价）
- [推断] 写入计费系统，扣减用户配额

## 4. 发送结束事件
通过 SSE 推送最终事件：
```
# 停止原因 + 用量
event: message_delta
data: {
  "type": "message_delta",
  "delta": { "stop_reason": "end_turn" },
  "usage": { "output_tokens": 800 }
}

# 流结束标记
event: message_stop
data: { "type": "message_stop" }
```

客户端收到 `message_stop` 后知道这次请求彻底完成。

## 5. HTTP 连接处理
SSE 流结束后的连接处理：
- 发送 HTTP chunked 的结束标记：`0\r\n\r\n`
- **Keep-Alive**：连接可能保持打开供后续请求复用
  - 减少 TCP 握手 + TLS 协商的开销
  - 超时后自动关闭
- **连接关闭**：如果服务端或客户端主动关闭
- 日志记录：请求耗时、token 数、stop_reason

## 6. KV Cache 决策
请求完成后，KV Cache 不一定立即释放：
- **保留策略**（期望用户继续对话）：
  - 整个 KV Cache 保留在显存中
  - 设置 TTL（约 5 分钟），超时自动释放
  - 下次请求如果前缀匹配，直接复用 → 省 Prefill
- **立即释放**（显存紧张）：
  - 释放 KV Cache 显存
  - 下次请求需要完整 Prefill
- [推断] **降级到 CPU**：
  - 从 GPU 显存移到 CPU 内存
  - 比完全释放慢一点恢复，但比重算快

```
KV Cache 生命周期：
  请求结束 → 保留(热) → 5分钟无访问 → 移到CPU(温) → 再过期 → 释放(冷)
```

## 7. GPU 计算资源释放
把 GPU 算力归还调度器：
- 当前请求从 Continuous Batching 的 batch 中移除
- 空出的"slot"可以被排队中的新请求占用
- 如果整个 batch 空了，GPU 进入等待状态

```
Batch 变化：
  请求完成前: [Req_A, Req_B, Req_C, Req_D]  ← 4 个并发 Decode
  Req_B 完成: [Req_A, _____, Req_C, Req_D]  ← 空出 1 个 slot
  新请求插入: [Req_A, Req_E, Req_C, Req_D]  ← 新请求 Prefill 后加入
```

## 8. 显存碎片整理
[推断] 定期整理 KV Cache 显存碎片：
- PagedAttention 使用分页机制，但仍可能产生碎片
- 已释放请求的 page 标记为可用
- 新请求可以使用这些回收的 page
- 如果碎片严重，可能需要 compaction（类似 GC）

```
显存布局：
  [Req_A pages][freed][Req_C pages][freed][freed][Req_D pages]
                 ↑ 碎片，新请求可使用这些空间
```

## 9. 请求日志 & 监控指标
记录请求的完整生命周期数据：
```
{
  "request_id": "req_abc123",
  "model": "claude-opus-4-6",
  "input_tokens": 12000,
  "output_tokens": 800,
  "cache_read_tokens": 8000,
  "stop_reason": "end_turn",
  "ttft_ms": 1200,           // Time To First Token
  "total_time_ms": 15000,    // 总耗时
  "tokens_per_second": 53,   // 生成速度
  "gpu_node": "gpu-rack3-node7",
  "queue_wait_ms": 200       // 排队等待时间
}
```

用于：
- 计费系统扣费
- 监控告警（延迟飙升、错误率上升）
- 性能分析（哪些请求慢、为什么慢）
- 容量规划（GPU 利用率趋势）

## 10. 准备接收下一个请求
一切清理完毕，系统回到就绪状态：
- 调度器更新该节点的负载信息（queue_depth-1, memory_available+N）
- 负载均衡器获知该节点有空闲资源
- 如果有排队请求 → 立即分配给该节点
- 如果队列为空 → GPU 空闲等待

```
完整生命周期：
  请求到达 → 排队 → Prefill → Decode → 停止 → 推送完成
       → Usage统计 → 日志 → KV Cache决策 → 资源释放 → 就绪
```

## 流程图

```
Token 生成 → 停止条件检测
  ↓
[1] 匹配停止条件（max_tokens / EOS / tool_use / stop_seq）
  ↓
[2] 最终 token 处理（是否包含在输出中）
  ↓
[3] Usage 统计（input/output/cache tokens）
  ↓
[4] 发送 message_delta + message_stop 事件
  ↓
[5] HTTP 连接处理（keep-alive / 关闭）
  ↓
[6] KV Cache 决策（保留 / 降级CPU / 释放）
  ↓
[7] GPU 计算资源释放（batch slot 空出）
  ↓
[8] 显存碎片整理（page 回收）
  ↓
[9] 请求日志 & 监控指标写入
  ↓
[10] 就绪 → 接收下一个请求
```
