# 请求路由调度：从 API Gateway 到 GPU 节点的 10 步

> 对应服务端推理流程第 1 步的深度拆解

## 1. TLS 终止 & HTTP 解析
客户端的 HTTPS 请求到达入口节点：
- TLS 握手 / 会话复用（Session Ticket）
- 解密请求体，解析 HTTP 头和 JSON Body
- 提取关键字段：`Authorization`、`anthropic-version`、`Content-Type`

## 2. API Key 鉴权
验证请求合法性：
- 解析 `x-api-key` 或 `Authorization: Bearer` 头
- 查鉴权服务：Key 是否有效、是否过期、是否被吊销
- 确认 Key 对应的 Organization / Workspace
- 检查 Key 的权限范围（某些 Key 可能只允许特定模型）

## 3. 配额 & 限流检查
防止滥用和过载：
- **RPM（Requests Per Minute）**：当前分钟请求数是否超限
- **TPM（Tokens Per Minute）**：预估 input_tokens + max_tokens 是否超配额
- **并发请求数**：当前该用户同时在跑多少请求
- 超限 → 返回 `429 Too Many Requests`，带 `retry-after` 头
- [推断] 使用分布式限流（Redis / 令牌桶算法）

## 4. 请求校验 & 预处理
验证请求体合法性：
- `model` 字段是否有效（claude-opus-4-6 / claude-sonnet-4-6 等）
- `max_tokens` 是否在允许范围内
- `messages` 格式是否合法（role 交替、content 类型）
- `tools` 定义是否符合 JSON Schema
- `thinking.budget_tokens` 是否合理
- 不合法 → 返回 `400 Bad Request`

## 5. 模型路由
根据 `model` 字段决定发往哪个集群：
- `claude-opus-4-6` → Opus GPU 集群（最大卡、最多卡）
- `claude-sonnet-4-6` → Sonnet GPU 集群（中等规模）
- `claude-haiku-4-5` → Haiku GPU 集群（最小最快）
- 每个集群可能分布在多个区域（us-east、us-west、eu-west）
- [推断] 优先选离用户近的区域，减少网络延迟

## 6. 负载均衡 & 节点选择
在目标集群内选择具体的推理节点：
- **节点健康检查**：排除宕机/过热/显存不足的节点
- **负载指标**：当前队列长度、GPU 利用率、显存占用
- **调度策略**：
  - 最少连接（Least Connections）
  - 加权轮询（Weighted Round Robin）
  - [推断] 考虑 KV Cache 亲和性（同一个用户的连续请求尽量发到同一节点，复用缓存）

## 7. 队列排队
选中节点后，请求进入该节点的等待队列：
- 如果 GPU 正忙（在处理其他请求的 Prefill / Decode）→ 排队等待
- **优先级队列**：[推断] 付费层级高的用户优先
- 如果队列满了 → 返回 `529 Overloaded`
- 等待时间就是用户感知到的"排队延迟"（高峰期可能几秒到几十秒）

## 8. Continuous Batching 调度
现代推理引擎的核心调度机制：
- 不是一个请求独占 GPU 直到完成，而是**动态组 batch**
- 正在 Decode 的请求腾出算力时，插入新请求的 Prefill
- 不同请求的 Decode 步骤合并成一个 batch 一起算
- **Iteration-level Scheduling**：每个 Decode 迭代重新决定 batch 组成
- 某个请求生成完毕 → 立即腾出位置给排队的新请求

## 9. 显存分配
为新请求分配 KV Cache 显存：
- 预估需要的 KV Cache 大小（input_tokens + 预期 output_tokens）
- [推断] PagedAttention 模式：按 block（如 16 tokens 一块）分配，不预占全部
- 显存不够 → 可能 swap 到 CPU 内存（慢但不拒绝）或拒绝请求
- 分配完成 → 请求正式进入推理流水线

## 10. 建立 Streaming 连接
请求准备就绪，建立响应通道：
- 设置 HTTP 响应头：`Content-Type: text/event-stream`、`Transfer-Encoding: chunked`
- 发送第一个 SSE 事件：`message_start`（包含 model、usage 初始信息）
- 连接建立后，后续每生成一个 token 就通过这个通道推送
- 设置超时监控：如果长时间无输出则断开

## 流程图

```
HTTPS 请求到达
  ↓
[1] TLS 终止 → HTTP 解析
  ↓
[2] API Key 鉴权（有效？权限？）
  ↓
[3] 限流检查（RPM/TPM/并发）→ 超限返回 429
  ↓
[4] 请求校验（model/messages/tools 格式）→ 非法返回 400
  ↓
[5] 模型路由（Opus/Sonnet/Haiku → 对应集群）
  ↓
[6] 负载均衡 → 选择最优 GPU 节点
  ↓
[7] 排队等待（队满返回 529）
  ↓
[8] Continuous Batching 调度（动态插入 batch）
  ↓
[9] 显存分配（KV Cache 空间）
  ↓
[10] 建立 SSE 连接 → message_start → 进入推理
```
